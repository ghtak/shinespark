## Phase 2 — Permission / Role CRUD (설계 중)

### Context

현재 `add_permission_to_role(role_id: i64, permission_id: i64)` 는 호출자가 정수 ID를 직접 알고 있어야 한다.
Permission/Role 자체를 생성·조회·삭제하는 관리 기능이 없으므로, 실제 어드민 페이지에서 사용하기 어렵다.
이 Phase에서는 `RbacUsecase` 안에 Permission/Role CRUD를 추가하고, 기존 링크 API를 code/name 기반 상위 레벨로 승격한다.

---

### 설계 결정 사항 (합의 필요)

#### 1. 기존 ID 기반 메서드 처리

현재:
```rust
async fn add_permission_to_role(&self, handle, role_id: i64, permission_id: i64) -> Result<()>;
async fn remove_permission_from_role(&self, handle, role_id: i64, permission_id: i64) -> Result<()>;
```

**옵션 A — 유지 + 상위 레벨 추가** (추천)
- 기존 ID 기반 메서드를 그대로 두고, 이름/코드 기반 편의 메서드를 추가
- `assign_permission_to_role(role_name, permission_code)` → 내부에서 ID 조회 후 기존 메서드 호출
- 기존 통합 테스트 변경 없음

**옵션 B — ID 기반 제거, 코드 기반으로 교체**
- 메서드 시그니처가 깔끔해지나 기존 테스트 수정 필요

#### 2. `delete_permission` 시 cascade 처리

마이그레이션에서 FK 제약이 주석 처리되어 있어 DB 레벨 cascade 없음.
→ usecase에서 직접 순서 제어:
1. `shs_iam_role_permission` 에서 해당 permission 행 삭제
2. `shs_iam_permission` 에서 삭제
3. 캐시 리로드

#### 3. `delete_role` 시 cascade 처리

동일한 이유로 usecase에서 순서 제어:
1. `shs_iam_user_role` 에서 해당 role 행 삭제
2. `shs_iam_role_permission` 에서 해당 role 행 삭제
3. `shs_iam_role` 에서 삭제
4. 캐시 리로드

### 추가될 API (RbacUsecase trait)

```rust
// ── Permission CRUD ──────────────────────────────────────────
async fn create_permission(
    &self, handle, command: CreatePermissionCommand,
) -> Result<Permission>;

async fn delete_permission(
    &self, handle, permission_id: i64,
) -> Result<()>;  // cascade + cache reload

async fn list_permissions(
    &self, handle,
) -> Result<Vec<Permission>>;

async fn find_permission_by_code(
    &self, handle, code: &str,
) -> Result<Option<Permission>>;

// ── Role CRUD ────────────────────────────────────────────────
async fn create_role(
    &self, handle, command: CreateRoleCommand,
) -> Result<Role>;

async fn delete_role(
    &self, handle, role_id: i64,
) -> Result<()>;  // cascade + cache reload

async fn list_roles(
    &self, handle,
) -> Result<Vec<Role>>;

// ── 상위 레벨 링크 (code/name 기반) ─────────────────────────
async fn assign_permission_to_role(
    &self, handle, role_name: &str, permission_code: &str,
) -> Result<()>;  // 내부: find_role_by_name + find_permission_by_code → add_permission_to_role

async fn revoke_permission_from_role(
    &self, handle, role_name: &str, permission_code: &str,
) -> Result<()>;  // 동일 패턴
```

    pub description: String,

### Repository 추가 메서드 (RbacRepository trait)

```rust
// Permission
async fn create_permission(&self, handle, code: &str, description: &str) -> Result<Permission>;
async fn delete_permission(&self, handle, id: i64) -> Result<()>;
async fn list_permissions(&self, handle) -> Result<Vec<Permission>>;
async fn find_permission_by_code(&self, handle, code: &str) -> Result<Option<Permission>>;
async fn delete_permissions_by_permission_id(&self, handle, permission_id: i64) -> Result<()>;
// └ shs_iam_role_permission 에서 permission_id 기준 삭제 (delete_permission cascade 용)

// Role
async fn create_role(&self, handle, name: &str, description: &str) -> Result<Role>;
async fn delete_role(&self, handle, id: i64) -> Result<()>;
async fn list_roles(&self, handle) -> Result<Vec<Role>>;
async fn delete_role_permissions_by_role_id(&self, handle, role_id: i64) -> Result<()>;
// └ shs_iam_role_permission 에서 role_id 기준 삭제 (delete_role cascade 용)
async fn delete_user_roles_by_role_id(&self, handle, role_id: i64) -> Result<()>;
// └ shs_iam_user_role 에서 role_id 기준 삭제 (delete_role cascade 용)
```

### 구현 순서 (체크리스트)

- [ ] **Step 1**: CQRS 구조체 추가 — `rbac_usecase.rs` 상단
- [ ] **Step 2**: `RbacRepository` trait에 메서드 10개 추가
- [ ] **Step 3**: SQL 파일 10개 생성
- [ ] **Step 4**: `SqlxRbacRepository` 구현 (RbacQuery variant 추가)
- [ ] **Step 5**: `RbacUsecase` trait에 메서드 추가
- [ ] **Step 6**: `DefaultRbacUsecase` 구현
  - `create_permission`: `AlreadyExists` 에러 처리 (`code` UNIQUE 제약)
  - `create_role`: 동일
  - `delete_permission`: 순서 제어 (role_permission 먼저 삭제 → permission 삭제 → cache reload)
  - `delete_role`: 순서 제어 (user_role + role_permission → role 삭제 → cache reload)
  - `assign_permission_to_role` / `revoke_permission_from_role`: code/name 조회 후 기존 ID 기반 메서드 위임
- [ ] **Step 7**: 통합 테스트 추가

### 에러 처리 정책

| 상황 | 에러 |
|------|------|
| 중복 `code`로 permission 생성 | `AlreadyExists` |
| 중복 `name`으로 role 생성 | `AlreadyExists` |
| 존재하지 않는 code/name으로 링크 요청 | `NotFound` |
| 이미 연결되지 않은 링크 해제 | 무시 (멱등) |
