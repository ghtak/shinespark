# RBAC Usecase 구현 플랜

## Context

`docs/04. rbac.md` 설계 문서를 기반으로 현재 비어있는 `RbacUsecase` trait과 `DefaultRbacUsecase` 구현체를 완성한다.
핵심은 서버 구동 시 DB에서 `role_id → permission_codes` 맵을 메모리에 적재하고, 이후 모든 권한 검사(`check_perm`)는 DB 접근 없이 동기 함수로 처리하는 것이다.
관리 작업(권한 추가/제거)이 발생할 때만 DB를 쓰고 캐시를 갱신한다.

---

## 구현 체크리스트

### Phase 1 — Repository 계층

- [ ] **Step 1**: `RbacRepository` trait 생성
  - 파일: `shinespark-identity/src/repositories/rbac_repository.rs`
  - 메서드: `load_role_permissions`, `find_role_by_name`, `assign_role_to_user`, `remove_role_from_user`, `add_permission_to_role`, `remove_permission_from_role`
  - 패턴: `&self`, `&mut Handle<'_>` — 기존 `UserRepository`와 동일

- [ ] **Step 2**: `repositories.rs` 에 re-export 추가
  - `mod rbac_repository; pub use rbac_repository::*;`

- [ ] **Step 3**: SQL 파일 6개 생성
  - 디렉토리: `shinespark-identity/sql/rbac_repository/`

  | 파일 | SQL |
  |------|-----|
  | `load_role_permissions.sql` | `SELECT rp.role_id, p.code FROM shs_iam_role_permission rp INNER JOIN shs_iam_permission p ON p.id = rp.permission_id` |
  | `find_role_by_name.sql` | `SELECT * FROM shs_iam_role WHERE name = $1` |
  | `assign_role_to_user.sql` | `INSERT INTO shs_iam_user_role (user_id, role_id) VALUES ($1, $2) ON CONFLICT (user_id, role_id) DO NOTHING` |
  | `remove_role_from_user.sql` | `DELETE FROM shs_iam_user_role WHERE user_id = $1 AND role_id = $2` |
  | `add_permission_to_role.sql` | `INSERT INTO shs_iam_role_permission (role_id, permission_id) VALUES ($1, $2) ON CONFLICT (role_id, permission_id) DO NOTHING` |
  | `remove_permission_from_role.sql` | `DELETE FROM shs_iam_role_permission WHERE role_id = $1 AND permission_id = $2` |

- [ ] **Step 4**: `SqlxRbacRepository` 구현체 생성
  - 파일: `shinespark-identity/src/infra/sqlx_rbac_repository.rs`
  - 파일 내부에 private `RbacQuery` enum 정의 (기존 `JwtQuery` 패턴 동일 — `include_str!` 매핑)
  - `execute()` vs `fetch_optional()`: DELETE는 `execute()`, INSERT/SELECT는 `fetch_optional()` / `fetch_all()` 사용

- [ ] **Step 5**: `infra.rs` 에 re-export 추가
  - `mod sqlx_rbac_repository; pub use sqlx_rbac_repository::*;`

### Phase 2 — Usecase 계층

- [ ] **Step 6**: `RbacUsecase` trait 확장
  - 파일: `shinespark-identity/src/usecases/rbac_usecase.rs`
  - 메서드:

  ```rust
  async fn load(&self, handle: &mut Handle<'_>) -> Result<()>;
  fn check_perm(&self, role_ids: &[i64], permission: &str) -> bool;  // 동기 — DB 접근 없음
  async fn assign_role_to_user(&self, handle: &mut Handle<'_>, user_id: i64, role_name: &str) -> Result<()>;
  async fn add_permission_to_role(&self, handle: &mut Handle<'_>, role_id: i64, permission_id: i64) -> Result<()>;
  async fn remove_permission_from_role(&self, handle: &mut Handle<'_>, role_id: i64, permission_id: i64) -> Result<()>;
  ```

  - `check_perm`은 일반 `fn` (async 아님). `#[async_trait]` 블록 안에 혼용 가능, trait object safe.

- [ ] **Step 7**: `DefaultRbacUsecase` 구현
  - 파일: `shinespark-identity/src/infra/default_rbac_usecase.rs`
  - 구조체:

  ```rust
  pub struct DefaultRbacUsecase<R: RbacRepository> {
      repository: Arc<R>,
      cache: Arc<std::sync::RwLock<HashMap<i64, HashSet<String>>>>,
  }
  ```

  - `load`: DB에서 `(role_id, code)` pairs 가져와 `HashMap<i64, HashSet<String>>` 재구성 후 캐시 교체
  - `check_perm` 와일드카드 로직:
    ```rust
    role_ids.iter().any(|id| {
        cache.get(id).map_or(false, |perms|
            perms.contains("*.*.all") || perms.contains(permission)
        )
    })
    ```
  - `add/remove_permission_to_role`: DB 쓰기 후 `self.load(handle).await?` 로 전체 캐시 리로드
    - 권한 변경은 관리자 작업(초희귀), 전체 리로드가 외과적 업데이트보다 단순하고 안전
  - `assign_role_to_user`: `find_role_by_name` → `NotFound` 에러 or `assign_role_to_user` repository 호출. 캐시 불변(role→perm 맵은 변경 없음)

### Phase 3 — Wiring & Seed

- [ ] **Step 8**: `seed_user.rs` 수정
  - 파일: `shinespark-identity/src/infra/seed_user.rs`
  - 시그니처 변경:
    ```rust
    pub async fn seed_admin(
        handle: &mut db::Handle<'_>,
        usecase: Arc<dyn UserUsecase>,
        rbac_usecase: Arc<dyn RbacUsecase>,
    )
    ```
  - 어드민 유저 생성/확인 후:
    ```rust
    let user_id = if let Some(u) = user { u.user.id } else { create_user(...).user.id };
    rbac_usecase.assign_role_to_user(handle, user_id, "admin").await.unwrap();
    ```
  - `assign_role_to_user`는 `ON CONFLICT DO NOTHING` 이므로 멱등 — 매 시작마다 호출 안전

- [ ] **Step 9**: `AppContainer` / `main.rs` 수정
  - 파일: `shinespark-app/src/main.rs`
  - `AppContainer::new()` 에:
    ```rust
    let rbac_repository = Arc::new(SqlxRbacRepository::new());
    let rbac_usecase = Arc::new(DefaultRbacUsecase::new(rbac_repository));
    ```
  - `main()` 에서 컨테이너 생성 후:
    ```rust
    container.rbac_usecase.load(&mut container.db.handle()).await.expect("rbac cache load failed");
    seed_admin(&mut container.db.handle(), container.user_usecase.clone(), container.rbac_usecase.clone()).await;
    ```
  - 순서: `load()` → `seed_admin()` (seed는 cache를 읽지 않으나 구동 순서 일관성 유지)

## 주요 설계 결정

| 항목 | 결정 | 이유 |
|------|------|------|
| RwLock 종류 | `std::sync::RwLock` | `check_perm`이 동기 fn — tokio RwLock 불필요 |
| 캐시 갱신 | 전체 리로드 (`load()` 재호출) | 권한 변경은 관리 이벤트(초희귀), 외과적 업데이트보다 단순 |
| 와일드카드 | `*.*.all` 하드코딩 리터럴 매칭 | 설계 문서 기준 단일 슈퍼어드민 패턴만 존재 |
| `RbacQuery` enum 위치 | `sqlx_rbac_repository.rs` 내부 private | 기존 `JwtQuery` 패턴 동일, 관심사 분리 |
| 캐시 불변 조건 | `assign_role_to_user`는 캐시 미갱신 | 캐시는 role→perm 맵; user→role 변경은 토큰 재발급 시 반영 |

## 검증 계획

1. `cargo build` — 컴파일 통과 확인 (`include_str!` 경로 포함)
2. 애플리케이션 기동 후 `shs_iam_user_role` 테이블에 `admin@shinespark.dev` → admin role 행 확인
3. `GET /identity/me` — admin 로그인 후 JWT claims의 `role_ids`에 admin role id 포함 확인
4. (통합 테스트) `check_perm(&[admin_role_id], "user.read.all")` → `true` (*.*.all 경유)
5. (통합 테스트) `check_perm(&[user_role_id], "user.update.own")` → `true`
6. (통합 테스트) `check_perm(&[user_role_id], "user.create.all")` → `false`
