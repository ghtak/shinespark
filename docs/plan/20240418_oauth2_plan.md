# OAuth2 Social Login 설계 계획

> 원본 요구사항: `docs/plan/20240418_oauth2.md`

---

## 목표

Google OAuth2 인증 흐름 구현. `{provider}/login` → redirect, `{provider}/callback` → code exchange → user 반환.  
GoogleLoginUsecase 만 구현, 미가입 시 자동 회원가입.

## Phase 2 — Contract

### Entity

`shinespark-identity/src/entities.rs` 에 추가:

```rust
#[derive(Debug, Clone, PartialEq, Eq, Serialize, Deserialize)]
pub enum OAuth2Provider {
    Google,
    Apple,
    Kakao,
}

impl OAuth2Provider {
    pub fn as_str(&self) -> &'static str { /* "google" | "apple" | "kakao" */ }
}
```

`InitialCredentials::Social` 의 `provider` 필드 타입 — 기존 `String` 유지 (provider_uid 와 함께 저장).

### SocialLoginUsecase trait

`shinespark-identity/src/usecases/social_login_usecase.rs`:

```rust
#[async_trait]
pub trait SocialLoginUsecase: Send + Sync {
    /// provider 의 OAuth2 authorization URL 반환
    async fn login(&self, cmd: SocialLoginCommand) -> Result<String>;

    /// code 교환 → user 반환 (없으면 자동 가입)
    async fn callback(
        &self,
        handle: &mut Handle<'_>,
        cmd: SocialCallbackCommand,
    ) -> Result<UserAggregate>;
}

pub struct SocialLoginCommand {
    pub state: String,   // CSRF 방어용 nonce
}

pub struct SocialCallbackCommand {
    pub code: String,
    pub state: String,
}
```

- `login` 은 외부 HTTP 없이 URL 조립만 — `Handle` 불필요
- `callback` 은 DB 접근(회원가입) 이 있으므로 `Handle` 필요

## Phase 3 — Implementation

### DefaultGoogleLoginUsecase

`shinespark-identity/src/infra/default_google_login_usecase.rs`:

```
pub struct DefaultGoogleLoginUsecase {
    config: GoogleLoginConfig,
    http: reqwest::Client,
    user_usecase: Arc<dyn UserUsecase>,
    login_usecase: Arc<dyn LoginUsecase>,
}
```

#### `login` 흐름

1. auth URL 포맷팅:
   ```
   https://accounts.google.com/o/oauth2/v2/auth
     ?response_type=code
     &client_id={client_id}
     &redirect_uri={redirect_uri}
     &scope={scope}
     &state={state}
     &access_type=offline
     &prompt=consent
   ```
2. URL 문자열 반환

#### `callback` 흐름

1. `POST https://oauth2.googleapis.com/token` (reqwest)
   - params: `code`, `client_id`, `client_secret`, `redirect_uri`, `grant_type=authorization_code`
   - 응답: `{ id_token, access_token, ... }`
2. `id_token` JWT decode (검증 없이 payload 만) → `sub` 추출 (Google 사용자 식별자)
3. `login_usecase.login(handle, LoginCommand::Social { provider: "google", provider_uid: sub })` 시도
   - `Ok(user)` → 반환
   - `Err(NotFound)` → 자동 가입 분기:
     - id_token payload 에서 `email`, `name` 추출
     - `user_usecase.create_user(handle, CreateUserCommand { ..., credentials: InitialCredentials::Social { provider: "google", provider_uid: sub } })` 호출
     - 생성된 user 반환

### 에러 처리

| 상황 | `shinespark::Error` |
|---|---|
| reqwest 실패 | `Internal` |
| id_token 디코드 실패 | `Internal` |
| `sub` 없음 | `IllegalState` |
| 회원가입 충돌 (이메일 중복) | `AlreadyExists` (LoginUsecase 위임) |

## Phase 4 — HTTP Layer

### 라우트

`shinespark-app/src/http/routes/identity/oauth2.rs`:

| Method | Path | 동작 |
|---|---|---|
| GET | `/identity/oauth2/:provider/login` | `SocialLoginCommand { state: uuid }` → 307 Redirect to Google |
| GET | `/identity/oauth2/:provider/callback` | query `?code=&state=` → `SocialCallbackCommand` → `UserAggregate` 반환 |

- `callback` 응답: `ApiResponse<UserAggregate>` (JWT 발급은 추후 확장)
- `:provider` 파싱 실패 시 `ApiError(NOT_FOUND, 400)`

핸들러 시그니처:

```rust
async fn login_handler(
    State(container): State<Arc<AppContainer>>,
    Path(provider): Path<String>,
) -> Result<Redirect, ApiError>

async fn callback_handler(
    State(container): State<Arc<AppContainer>>,
    Path(provider): Path<String>,
    Query(params): Query<CallbackParams>,
    // DB 핸들은 Extension 또는 풀에서 직접 획득 (현재 패턴 확인 필요)
) -> ApiResult<UserResponse>
```

### AppContainer 조립

`shinespark-app/src/main.rs`:
```rust
pub struct AppContainer {
    // 기존 필드 ...
    pub google_login_usecase: Arc<dyn SocialLoginUsecase>,
}
```

조립 순서:
1. `GoogleLoginConfig` 로드
2. `reqwest::Client::new()`
3. `DefaultGoogleLoginUsecase::new(config, http, user_usecase.clone(), login_usecase.clone())`
4. `Arc::new(...)` → `google_login_usecase`

## 리스크 및 주의

| 항목 | 내용 |
|---|---|
| `state` 검증 | 현재 계획에서 `state` 는 클라이언트가 전달하는 값이며 서버측 저장이 없음. CSRF 방어 강화 필요시 Redis/session 저장 고려 |
| id_token 서명 검증 생략 | `sub` 식별용으로만 사용, 서명 검증 없음 — Google 공개키 검증은 추후 보안 강화 시 추가 |
| `callback` 의 Handle 획득 | 현재 handler 패턴에서 DB Handle 을 직접 받는 방식 확인 필요 (Extension 또는 pool에서 begin) |
| Apple/Kakao | enum 만 선언, 구현은 제외 — 라우트에서 `Google` 외 provider 는 `NotImplemented` 반환 |
