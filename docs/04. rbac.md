# 권한 제어 (RBAC - Role Based Access Control) 시스템 설계 가이드

이 문서는 Shinespark 시스템의 내부 구성요소 보호 및 접근 제어를 위한 RBAC 아키텍처 및 구현 설계 방향을 정의합니다.

## 1. 개요 및 디자인 철학

접근 제어의 복잡성을 낮추기 위해 **"신원 제어(Identity)"**와 **"접근 제어(Authorization)"** 도메인을 완벽히 분리합니다.

- **`UserUsecase`**: 순수한 사용자 정보(Profile), 활성화 상태 등 계정 단위의 라이프사이클(CRUD)에 집중하는 단일 책임을 가집니다.
- **`RbacUsecase`**: 모든 권한(Permission), 역할(Role), 및 유저에 대한 역할 부여/해제 로직을 전담합니다.

## 2. 역할(Role)과 권한(Permission)의 분리

* **권한 (Permission)**: `user.read`, `dashboard.view`와 같이 잘게 쪼개진 단위입니다. 코드 베이스의 상수나 Enum으로 정의되어 각 기능 진입 시 필수 요건으로 검증(`check_perm`)됩니다.
* **역할 (Role)**: `관리자`, `일반사용자`, `협력사` 등 추상적인 비즈니스 단위 뭉치입니다. 시스템 관리자는 권한 코드가 무엇인지 구체적으로 알 필요 없이, 사전에 정의된 '역할'에 필요한 '권한'들을 단순 매핑해 주면 됩니다.

> 💡 **예시**: 만약 특정 파트너사에게 관리자 대시보드 접근 권한을 임시로 주고자 한다면, 코드를 수정하고 배포할 필요 없이 어드민 페이지(DB 수정)에서 '파트너' 역할(Role)에 'dashboard.view' 권한(Permission)을 하나 덧붙여 연결(`shs_iam_role_permission` 매핑)하기만 하면 즉시 통과됩니다!

## 3. 엔티티 구조 및 DTO (Read Model)

데이터베이스에서 무거운 조인(JOIN) 객체를 들고 오는 대신 실리적인 데이터 직렬화를 사용합니다.

```rust
// 1. 단일 테이블(shs_iam_user) 매핑 객체 - 순수 Entity
pub struct User { ... }

// 2. 권한 정보가 포함된 복합 객체 - 읽기 전용 DTO
#[derive(Debug, Clone, Serialize, sqlx::FromRow)]
pub struct UserWithRoles {
    #[sqlx(flatten)]
    pub user: User,

    // SQL 상에서 `json_agg()`로 합쳐진 role_id 숫자들의 배열
    // 무거운 Role 덩어리를 파싱하지 않고 ID 식별자만 가져와 통신 효율 향상
    pub role_ids: sqlx::types::Json<Vec<u64>>,
}
```

## 4. 인메모리 글로벌 캐시 (Global Cache) 아키텍처

인가(Authorization) 처리가 필요한 API는 사용자의 **무수히 많은 모든 HTTP 요청마다 수시로 발생**합니다. 이를 매번 DB 풀에서 조회하는 것은 시스템 성능의 치명적인 병목이 됩니다.

**`RbacService`** 는 이를 극복하기 위해 글로벌(단일 애플리케이션) 캐시를 운용합니다.

1. **초기/조회 구조**: 서버 구동 시(혹은 최초 요청 시점), `RbacService`는 DB에서 `role_id`에 속한 `Permission`들의 데이터를 가져와 메모리 캐시(`Arc<RwLock<HashMap<u64, Vec<Permission>>>>` 혹은 `arc-swap`, `dashmap` 등)에 안전하게 가지고 있습니다.
2. **동기화 전략**: 권한 매핑 정보가 바뀔 일이 있을 때만(일반적으로 어드민 페이지에서의 작업 발생) 권한 통제를 전담하는 `RbacService.add_permission_to_role(...)` 메서드 내부에서 DB를 업데이트 한 직후 로컬 캐시를 갱신해 줍니다.
3. **권한 검증 로직 (`check_perm`)**: 유저의 로그인 정보(토큰 등)에서 파싱된 `role_ids`(예: `[1, 5]`)를 기반으로, 데이터베이스 쿼리 비용 0, 네트워크 지연 0 상태로 내부 메모리에서 즉각적으로 권한 소유 여부를 판별(`has_permission(&role_ids, "user.update.own")`) 후 응답합니다.

## 5. 권한 명명 규칙: 리소스.액션.스코프 (Resource.Action.Scope)

유연하고 확장 가능한 권한 제어를 위해 모든 Permission Code는 `리소스.액션.스코프` 구조를 따릅니다. 동일한 행위라도 사용자가 어떤 범위(Scope)의 데이터에 접근할 수 있는지 명확히 구분하여 응용 계층의 인가(Authorization) 로직을 일관되게 유지합니다.

### 5.1. 구성 요소
* **Resource**: 제어의 대상이 되는 도메인 객체 (예: `user`, `post`, `billing`)
* **Action**: 수행하려는 행위 (예: `read`, `create`, `update`, `delete`)
* **Scope**: 접근이 허용되는 데이터의 소유권 또는 범위 (예: `all`, `group`, `own`)

### 5.2. 스코프 (Scope) 종류
1. **`all` (전체 범위)**
   * **의미**: 시스템 내 등록된 해당 리소스 전체에 대한 접근을 무조건 허용합니다. 주로 최고 관리자(Admin) 또는 시스템 관리 시스템에 부여됩니다.
   * **예시**: `user.read.all` (모든 사용자 정보 조회 가능), `*.*.all` (시스템 전체 모든 권한)
2. **`group` (그룹/조직 범위)**
   * **의미**: 사용자가 속한 특정 그룹, 부서, 조직 내의 리소스에 한해 접근을 허용합니다. B2B SaaS의 워크스페이스 관리자나 특정 팀의 팀장 레벨에게 부여됩니다.
   * **예시**: `user.read.group` (같은 부서/조직원 정보만 조회 가능), `post.update.group` (팀 게시판 글 수정)
3. **`own` (본인 소유 범위)**
   * **의미**: 사용자 본인이 직접 소유하거나 본인과 연관된 리소스에 대해서만 접근을 허용합니다. 일반적인 B2C 사용자 환경에서 기본적으로 사용됩니다.
   * **예시**: `user.update.own` (본인 프로필만 수정), `post.delete.own` (자신이 작성한 글만 삭제 가능)

### 5.3. 응용 로직 구조 활용 (`check_perm`) 패턴
시스템 내부적으로 데이터 소유권을 확인한 후, 그 범위에 맞는 권한 유무를 체크하는 식으로 적용됩니다.

```rust
// 예시: 사용자 정보 갱신(Update) 권한 검증 시나리오
if is_owner {
    // 1. 본인이 본인 데이터를 조작할 때
    has_permission(&role_ids, "user.update.own") || has_permission(&role_ids, "user.update.all")
} else if is_same_group {
    // 2. 관리자가 같은 조직 내 다른 유저의 데이터를 조작할 때
    has_permission(&role_ids, "user.update.group") || has_permission(&role_ids, "user.update.all")
} else {
    // 3. 전혀 다른 조직의 데이터를 조작할 때 (전체 권한 관리자만 가능)
    has_permission(&role_ids, "user.update.all")
}
```

위와 같이 규칙을 강제함으로써, 역할 개수가 수십 개로 늘어나고 리소스 종류가 방대해져도 각 역할에 권한을 단순 삽입하는 것만으로 안전하고 치밀하게 통제할 수 있습니다.
